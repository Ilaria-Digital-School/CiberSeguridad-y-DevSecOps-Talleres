# Taller: colocación de pruebas de seguridad en un flujo de trabajo existente

**Recordatorio teórico** Integrar pruebas de seguridad en un flujo de trabajo existente es el corazón del enfoque DevSecOps. La idea central es que la seguridad no debe ser un paso final, sino una capa transversal presente en cada etapa del ciclo de desarrollo: desde el primer commit hasta el despliegue en producción. Este concepto se resume con el principio Shift-Left: detectar vulnerabilidades lo antes posible, cuando el coste de corrección es mínimo. Existen tres grandes familias de pruebas de seguridad que conviene dominar: el SAST (Static Application Security Testing), que analiza el código fuente sin ejecutarlo; el DAST (Dynamic Application Security Testing), que ataca la aplicación en ejecución como lo haría un atacante externo; y el SCA (Software Composition Analysis), que identifica vulnerabilidades conocidas en las dependencias de terceros. A estos se suman las políticas de ramas seguras y los hooks pre-commit, ya vistos en talleres anteriores. El error frecuente del principiante es añadir todas las pruebas de golpe, bloqueando el pipeline por completo y generando resistencia en el equipo. El enfoque correcto es incremental: se añade una prueba a la vez, se valida que no rompe el flujo existente, y se ajusta el umbral de fallo de manera progresiva. Mini-ejemplo mental: imagine un pipeline CI/CD con tres etapas (build → test → deploy).
