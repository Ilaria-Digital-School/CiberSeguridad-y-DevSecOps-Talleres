# Taller: simular un secreto cometido y corregirlo

**RECORDATORIO TEÓRICO** En el contexto de DevSecOps, un 'secreto' (secret) es cualquier dato sensible que otorga acceso a un recurso: contraseñas, tokens de API, claves SSH, credenciales de bases de datos, certificados privados, etc. Uno de los errores más frecuentes y peligrosos en el desarrollo de software es comprometer (commit) accidentalmente un secreto en un repositorio Git, ya sea público o privado. Una vez que un secreto ha sido registrado en el historial de Git, simplemente borrarlo del código fuente no es suficiente: el historial de commits conserva una copia permanente y accesible. El modelo mental que debes interiorizar es el siguiente: Git es una máquina del tiempo. Todo lo que alguna vez fue confirmado queda grabado, incluso si luego se modifica o elimina el archivo. Los atacantes explotan herramientas como truffleHog, git-secrets o simplemente `git log` para rastrear credenciales expuestas en repositorios públicos de GitHub, GitLab o Bitbucket. Se han producido brechas de seguridad masivas por este motivo. Errores frecuentes: subir un archivo `.env` con credenciales reales, hardcodear una API key directamente en el código, olvidar añadir archivos sensibles al `.gitignore` antes del primer commit. La corrección correcta implica: revocar el secreto expuesto, reescribir el historial de Git para eliminarlo, y adoptar una gestión segura de secretos (variables de entorno, gestores como Vault o el sistema de secrets del CI/CD). Mini-ejemplo: un desarrollador añade `API_KEY=abc123secret` en su archivo `config.py`, hace commit y push. Aunque luego borra la línea y vuelve a hacer commit, cualquiera puede ejecutar `git log -p`...
