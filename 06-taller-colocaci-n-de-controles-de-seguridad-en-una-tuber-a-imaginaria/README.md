# Taller: colocación de controles de seguridad en una tubería imaginaria

RECORDATORIO TEÓRICO Antes de comenzar el taller, repasemos los conceptos esenciales vistos en los capítulos anteriores. Una canalización CI/CD (Integración Continua / Despliegue Continuo) es el conjunto de etapas automatizadas que transforma el código fuente de un desarrollador en una aplicación desplegada en producción. En el enfoque DevSecOps, la seguridad no se añade al final del proceso, sino que se integra en cada etapa de esta canalización, siguiendo el principio de 'Shift Left' (desplazar la seguridad hacia la izquierda, es decir, hacia las fases más tempranas del desarrollo). Los tres grandes tipos de controles de seguridad que se pueden insertar en una tubería son: SAST (análisis estático del código fuente, antes de que el código se ejecute), DAST (análisis dinámico sobre una aplicación en funcionamiento) y SCA (análisis de las dependencias y librerías de terceros). A estos se suman controles como el análisis de secretos expuestos, el escaneo de contenedores y las pruebas de calidad con impacto en seguridad. El error más frecuente de los principiantes es creer que basta con colocar un único control al final de la tubería, justo antes del despliegue. Esto es contraproducente: cuanto más tarde se detecta una vulnerabilidad, más caro y complejo es corregirla. Otro error habitual es confundir SAST con DAST: el primero analiza el código sin ejecutarlo (como leer una receta), el segundo prueba la aplicación en vivo (como probar el plato ya cocinado).
