# Taller de : adaptación de una canalización segura a un proyecto de comercio electrónico

**RECORDATORIO TEÓRICO** Una canalización CI/CD (Integración Continua / Entrega Continua) es el conjunto de etapas automatizadas que permiten pasar del código fuente a una aplicación desplegada en producción. En un enfoque DevSecOps, la seguridad no es un paso final, sino una capa integrada en cada etapa del pipeline. Los tres pilares que guían esta integración son la Confidencialidad, la Integridad y la Disponibilidad (tríada CIA), ya estudiados en módulos anteriores. Los controles de seguridad más habituales en una canalización son: el SAST (análisis estático del código antes de su ejecución), el SCA (análisis de dependencias y bibliotecas de terceros), el DAST (pruebas dinámicas contra la aplicación en ejecución) y el análisis de imágenes de contenedores. Cada control se aplica en un momento preciso del pipeline: SAST y SCA en la fase de build, DAST en la fase de staging o pre-producción. Un error frecuente de los principiantes es pensar que basta con añadir un único control al final del pipeline para «tener seguridad». Esto es incorrecto: una vulnerabilidad detectada tarde cuesta mucho más en tiempo y recursos que una detectada en la fase de desarrollo. Mini-ejemplo: si un desarrollador introduce una dependencia con una vulnerabilidad conocida (CVE), un SCA configurado en la etapa de build lo detectará antes de que el código llegue a producción. --- **ENUNCIADO DEL TALLER** En este taller trabajarás de forma autónoma sobre un escenario concreto: una empresa ficticia llamada ShopSafe lanza una plataforma de comercio electrónico.
