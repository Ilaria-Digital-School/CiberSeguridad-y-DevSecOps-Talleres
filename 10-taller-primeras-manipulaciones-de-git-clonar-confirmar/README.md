# Taller: primeras manipulaciones de Git (clonar, confirmar)

**Recordatorio teórico** Git es un sistema de control de versiones distribuido que permite registrar el historial de cambios de un proyecto, colaborar con otros desarrolladores y mantener trazabilidad completa sobre quién modificó qué y cuándo. En el contexto de DevSecOps, Git no es solo una herramienta de desarrollo: es la base de toda la cadena CI/CD y el punto de partida desde donde se desencadenan los análisis de seguridad automatizados (SAST, SCA, etc.). Conceptos clave que debes tener claros antes de empezar: un **repositorio** (repo) es el contenedor del proyecto versionado; **clonar** (git clone) significa copiar un repositorio remoto a tu máquina local; un **commit** es una instantánea del estado del proyecto en un momento dado, acompañada de un mensaje descriptivo; el **staging area** (git add) es la zona donde seleccionas los cambios que quieres incluir en el próximo commit. Error frecuente: confundir «guardar un archivo» con «hacer un commit». Guardar solo modifica el disco local; el commit registra ese cambio en el historial de Git. Otro error común es escribir mensajes de commit vagos como «cambios» o «update», lo que dificulta la auditoría de seguridad posterior. Mini-ejemplo mental: imagina que Git es un cuaderno de bitácora de un barco. Cada entrada (commit) describe exactamente qué pasó, cuándo y quién lo escribió. Sin ese registro, sería imposible reconstruir qué cambió entre una versión segura y una vulnerada. --- **Enunciado del taller** Este taller marca tu primer contacto real con Git en un entorno de trabajo DevSecOps.
