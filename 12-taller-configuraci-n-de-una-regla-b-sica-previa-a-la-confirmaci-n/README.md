# Taller: configuración de una regla básica previa a la confirmación

RECORDATORIO TEÓRICO Un hook de Git (gancho) es un script que Git ejecuta automáticamente antes o después de ciertos eventos, como un commit, un push o un merge. El hook 'pre-commit' se ejecuta justo antes de que Git registre un commit: si el script devuelve un código de salida distinto de 0, el commit es rechazado. Esta capacidad es fundamental en DevSecOps porque permite detectar problemas de seguridad en el momento más temprano posible del ciclo de desarrollo, antes de que el código llegue al repositorio remoto. Modelo mental clave: piensa en el hook pre-commit como un guardia de seguridad en la puerta de entrada del repositorio. Nadie puede entrar (hacer commit) sin pasar el control. Si el guardia detecta algo sospechoso (un secreto, una contraseña en duro, un token de API), bloquea el paso. Errores frecuentes: (1) Olvidar dar permisos de ejecución al script del hook con 'chmod +x'; (2) confundir la ubicación del archivo, que debe estar en '.git/hooks/pre-commit' (sin extensión); (3) asumir que el hook se comparte automáticamente con el equipo vía Git (los hooks NO se versifican en '.git', hay que distribuirlos manualmente o usar herramientas como 'pre-commit framework'). Mini-ejemplo: un script pre-commit que busca la cadena 'password=' en los archivos staged y rechaza el commit si la encuentra es suficiente para prevenir uno de los errores más comunes y costosos en desarrollo: exponer credenciales en el código fuente. --- ENUNCIADO DEL TALLER En el capítulo anterior estudiaste las políticas de ramas seguras y la trazabilidad...
